> EC2 OVERVIEW

Amazon Elastic Compute Cloud (EC2) provides scalable computing capacity in the AWS cloud. Think of it as renting virtual servers on demand.

EC2 eliminates the need to invest in hardware upfront, allowing you to develop and deploy applications faster. You can launch as many or as few virtual servers as needed.

Core Concepts:

  • Instance: A virtual server in the cloud
  • AMI: Amazon Machine Image - template for your instance
  • Instance Type: Hardware configuration (CPU, memory, storage, networking)
  • Region: Physical location of AWS data centers
  • Availability Zone: Isolated locations within regions
  • Security Group: Virtual firewall for your instance
  • Key Pair: Secure login credentials

EC2 Instance Lifecycle:

    [AMI] → [Launch] → [Pending] → [Running]
                          ↓            ↓
                      [Terminated] ← [Stopping] ← [Stopped]
                                         ↓
                                    [Rebooting]

Why EC2?

  • Complete control over computing resources
  • Scale up or down in minutes
  • Pay only for capacity you use
  • Choose from various instance types optimized for different use cases
  • Integrate with other AWS services seamlessly

> EBS (ELASTIC BLOCK STORE)

Persistent block storage for EC2 instances - your virtual hard drives in the cloud.

EBS Volume Types:

gp3 (General Purpose SSD)

  • 3,000 IOPS baseline
  • 125 MB/s throughput
  • Can provision up to 16,000 IOPS
  • Most cost-effective for most workloads
--volume-type gp3 --iops 3000

gp2 (Previous Gen SSD)

  • 3 IOPS per GB (min 100, max 16,000)
  • Burstable to 3,000 IOPS
  • Legacy - migrate to gp3
--volume-type gp2

io2 (Provisioned IOPS SSD)

  • Up to 64,000 IOPS
  • 99.999% durability
  • Critical business applications
  • Large databases
--volume-type io2 --iops 50000

st1 (Throughput Optimized HDD)

  • Big data, data warehouses
  • Log processing
  • Cannot be boot volume
  • Max 500 MB/s throughput
--volume-type st1

sc1 (Cold HDD)

  • Lowest cost storage
  • Infrequently accessed data
  • Cannot be boot volume
  • Max 250 MB/s throughput
--volume-type sc1

EBS Operations:

Creating and Attaching Volumes:

# Create volume aws ec2 create-volume \ --availability-zone us-east-1a \ --size 100 \ --volume-type gp3 \ --tag-specifications 'ResourceType=volume,Tags=[{Key=Name,Value=DataVolume}]' # Attach to instance aws ec2 attach-volume \ --volume-id vol-12345 \ --instance-id i-12345 \ --device /dev/sdf

Mounting in Linux:

# Check available disks lsblk # Create filesystem sudo mkfs -t xfs /dev/xvdf # Mount volume sudo mkdir /data sudo mount /dev/xvdf /data # Persistent mount echo "/dev/xvdf /data xfs defaults,nofail 0 2" | sudo tee -a /etc/fstab

EBS Snapshots:

Snapshots are incremental backups stored in S3. First snapshot contains all data, subsequent snapshots only store changes.
# Create snapshot aws ec2 create-snapshot \ --volume-id vol-12345 \ --description "Daily backup $(date +%Y-%m-%d)" # Create volume from snapshot aws ec2 create-volume \ --snapshot-id snap-12345 \ --availability-zone us-east-1b

EBS Encryption:

Always encrypt sensitive data! EBS encryption has no performance impact.
# Enable encryption by default aws ec2 enable-ebs-encryption-by-default # Create encrypted volume aws ec2 create-volume \ --encrypted \ --kms-key-id arn:aws:kms:us-east-1:123456789012:key/12345

EBS vs Instance Store:

Feature EBS Instance Store
Persistence Data persists Data lost on stop/terminate
Backup Snapshots supported No backup feature
Performance Good, consistent Very high, variable
Use Case Databases, file systems Temporary storage, cache

> EC2 NETWORKING

Understanding EC2 networking is crucial for building secure, scalable architectures.

Network Components:

    Internet Gateway
           |
        [VPC]
           |
    +------+------+
    |             |
 Public       Private
 Subnet       Subnet
    |             |
[EC2-Web]    [EC2-DB]
    |             |
  [EIP]      [NAT Gateway]

IP Addressing:

Private IP:

  • Automatically assigned from subnet CIDR
  • Retained when instance stops
  • Used for internal VPC communication

Public IP:

  • Optionally assigned at launch
  • Released when instance stops
  • Changes after stop/start

Elastic IP (EIP):

  • Static public IP address
  • Can be reassigned between instances
  • Charged when not associated
# Allocate Elastic IP aws ec2 allocate-address --domain vpc # Associate with instance aws ec2 associate-address \ --instance-id i-12345 \ --allocation-id eipalloc-12345

Elastic Network Interfaces (ENI):

Virtual network cards that can be attached/detached from instances
# Create ENI aws ec2 create-network-interface \ --subnet-id subnet-12345 \ --description "Secondary network interface" \ --groups sg-12345 # Attach to instance aws ec2 attach-network-interface \ --network-interface-id eni-12345 \ --instance-id i-12345 \ --device-index 1

Enhanced Networking:

SR-IOV (Single Root I/O Virtualization):

  • Higher bandwidth
  • Lower latency
  • Lower CPU utilization
# Enable enhanced networking aws ec2 modify-instance-attribute \ --instance-id i-12345 \ --ena-support

Placement Groups:

Cluster

Low latency, high throughput

Same rack, same AZ

Spread

High availability

Different racks

Partition

Large distributed workloads

Logical partitions

Network Performance Optimization:

  • Use instances with enhanced networking support
  • Place instances in same placement group for low latency
  • Use dedicated hosts for consistent performance
  • Monitor network metrics in CloudWatch
For network-intensive applications, choose instances with "Up to 25 Gbps" or higher network performance!

> SECURITY GROUPS

Security groups act as virtual firewalls controlling traffic to your instances.

Security groups are stateful - if you allow inbound traffic, the response is automatically allowed.

Security Group Rules:

Inbound Rules:

# SSH access (restrict source IP!) aws ec2 authorize-security-group-ingress \ --group-id sg-12345 \ --protocol tcp \ --port 22 \ --cidr 203.0.113.0/24 # HTTP from anywhere aws ec2 authorize-security-group-ingress \ --group-id sg-12345 \ --protocol tcp \ --port 80 \ --cidr 0.0.0.0/0 # MySQL from another security group aws ec2 authorize-security-group-ingress \ --group-id sg-12345 \ --protocol tcp \ --port 3306 \ --source-group sg-67890

Common Port References:

Service Port Protocol Use Case
SSH 22 TCP Linux administration
RDP 3389 TCP Windows administration
HTTP 80 TCP Web traffic
HTTPS 443 TCP Secure web traffic
MySQL 3306 TCP Database access
PostgreSQL 5432 TCP Database access
Redis 6379 TCP Cache access

Security Best Practices:

  • Never allow 0.0.0.0/0 on SSH (port 22)
  • Use specific IP ranges or security groups
  • Implement defense in depth with NACLs
  • Regularly audit security group rules
  • Use Systems Manager Session Manager instead of SSH
  • Enable VPC Flow Logs for traffic analysis

Troubleshooting Connectivity:

# Check security group rules aws ec2 describe-security-groups --group-ids sg-12345 # Test connectivity telnet instance-ip 80 # Check instance status aws ec2 describe-instance-status --instance-id i-12345
Can't connect? Check: 1) Security Group rules, 2) NACL rules, 3) Route tables, 4) Instance status

> KEY PAIRS & SSH ACCESS

Secure access to your EC2 instances using SSH key pairs.

Creating Key Pairs:

Using AWS Console/CLI:

# Create key pair aws ec2 create-key-pair \ --key-name MyKeyPair \ --query 'KeyMaterial' \ --output text > MyKeyPair.pem # Set correct permissions (Linux/Mac) chmod 400 MyKeyPair.pem

Using SSH-Keygen:

# Generate key pair ssh-keygen -t rsa -b 4096 -f MyKeyPair # Import public key to AWS aws ec2 import-key-pair \ --key-name MyKeyPair \ --public-key-material fileb://MyKeyPair.pub

Connecting to Instances:

Linux Instances:

# Basic SSH connection ssh -i MyKeyPair.pem ec2-user@public-ip-address # SSH with verbose output (debugging) ssh -vvv -i MyKeyPair.pem ec2-user@public-ip-address # SSH config file (~/.ssh/config) Host myserver HostName public-ip-address User ec2-user IdentityFile ~/keys/MyKeyPair.pem ServerAliveInterval 60

Default Usernames by AMI:

AMI Type Default Username
Amazon Linux 2 ec2-user
Ubuntu ubuntu
RHEL ec2-user or root
CentOS centos
Debian admin
SUSE ec2-user

Windows Instances:

# Get Windows password aws ec2 get-password-data \ --instance-id i-12345 \ --priv-launch-key MyKeyPair.pem

SSH Tunneling:

Use SSH tunnels to securely access private resources through a bastion host
# Port forwarding (access RDS through bastion) ssh -i MyKeyPair.pem -L 3306:rds-endpoint:3306 ec2-user@bastion-ip # Dynamic port forwarding (SOCKS proxy) ssh -i MyKeyPair.pem -D 8080 ec2-user@bastion-ip # Jump through bastion to private instance ssh -i MyKeyPair.pem -J ec2-user@bastion-ip ec2-user@private-ip

EC2 Instance Connect:

Browser-based SSH without managing keys! Available for Amazon Linux 2 and Ubuntu.
# Enable EC2 Instance Connect sudo yum install ec2-instance-connect # Connect using AWS CLI aws ec2-instance-connect send-ssh-public-key \ --instance-id i-12345 \ --availability-zone us-east-1a \ --instance-os-user ec2-user \ --ssh-public-key file://my-public-key.pub

Systems Manager Session Manager:

No SSH keys needed! Access instances through AWS Console or CLI without opening port 22.
# Start session aws ssm start-session --target i-12345 # Port forwarding through Session Manager aws ssm start-session \ --target i-12345 \ --document-name AWS-StartPortForwardingSession \ --parameters '{"portNumber":["3306"],"localPortNumber":["3306"]}'

> AMIs & SNAPSHOTS

Amazon Machine Images (AMIs) are templates for your EC2 instances.

AMI Components:

  • Root volume template (OS, applications)
  • Launch permissions
  • Block device mapping
  • Architecture (x86 or ARM)
  • Virtualization type (HVM or PV)

Creating Custom AMIs:

# Prepare instance (on the instance) # Remove sensitive data sudo rm -rf /home/*/.ssh/authorized_keys sudo rm -rf /root/.ssh/authorized_keys sudo rm -rf /var/log/* # Clear history history -c
# Create AMI from running instance aws ec2 create-image \ --instance-id i-12345 \ --name "MyWebServer-v1.0" \ --description "Web server with Apache and PHP" \ --no-reboot

AMI Best Practices:

  • Remove credentials and secrets before creating AMI
  • Use consistent naming convention
  • Tag AMIs with version, date, purpose
  • Test AMIs before production use
  • Regularly update base AMIs
  • Copy AMIs to multiple regions for DR

Sharing AMIs:

# Share with specific account aws ec2 modify-image-attribute \ --image-id ami-12345 \ --launch-permission "Add=[{UserId=123456789012}]" # Make public (careful!) aws ec2 modify-image-attribute \ --image-id ami-12345 \ --launch-permission "Add=[{Group=all}]"

AMI Lifecycle:

    [Running Instance]
            |
      Create Image
            |
       [Creating]
            |
       [Available] ← Register from Snapshot
            |
     [Deregistered]

Golden AMI Strategy:

Create a "golden" base AMI with your standard configuration, security updates, and monitoring agents.
# Automation with Systems Manager aws ssm create-document \ --name "GoldenAMIAutomation" \ --document-type "Automation" \ --content file://golden-ami-automation.json

AMI Costs:

  • No charge for AMI itself
  • Charged for underlying EBS snapshots
  • Snapshots are incremental
  • Delete unused AMIs to save costs

> EC2 PRICING MODELS

Choose the right pricing model to optimize costs.

Pricing Options Comparison:

Model Discount Commitment Use Case
On-Demand 0% None Variable workloads, testing

On-Demand Instances:

  • Pay by the second (Linux) or hour (Windows)
  • No upfront payment
  • No long-term commitment
  • Ideal for unpredictable workloads

Savings Plans:

Commit to consistent usage ($/hour) for 1 or 3 years. More flexible than Reserved Instances.

Spot Instances:

Spot instances can be terminated with 2-minute warning when AWS needs capacity!
# Request Spot Instance aws ec2 request-spot-instances \ --spot-price "0.03" \ --instance-count 1 \ --type "one-time" \ --launch-specification file://spot-spec.json

Spot Best Practices:

  • Use Spot Fleet for diversification
  • Handle interruptions gracefully
  • Mix with On-Demand instances
  • Use for stateless applications
  • Implement checkpointing for long jobs

Cost Optimization Strategies:

  1. Right-size instances: Use CloudWatch metrics to identify underutilized instances
  2. Use Auto Scaling: Scale down during off-peak hours
  3. Leverage Spot: For fault-tolerant workloads
  4. Purchase Savings Plans: For predictable usage
  5. Schedule instances: Stop development instances at night
Use AWS Compute Optimizer to get right-sizing recommendations based on your actual usage!

> MONITORING & METRICS

Monitor your EC2 instances to ensure optimal performance and availability.

CloudWatch Metrics:

Default EC2 Metrics (5-minute intervals):

  • CPU Utilization: Percentage of allocated compute units used
  • Network In/Out: Bytes received/sent on all interfaces
  • Disk Read/Write Ops: Operations on all instance store volumes
  • Status Checks: System and instance reachability

Detailed Monitoring (1-minute intervals):

# Enable detailed monitoring aws ec2 monitor-instances --instance-ids i-12345 # Get CPU metrics aws cloudwatch get-metric-statistics \ --namespace AWS/EC2 \ --metric-name CPUUtilization \ --dimensions Name=InstanceId,Value=i-12345 \ --statistics Average \ --start-time 2024-01-01T00:00:00Z \ --end-time 2024-01-01T01:00:00Z \ --period 300

Custom Metrics:

Default metrics don't include memory or disk space. Use CloudWatch Agent for these!
# Install CloudWatch Agent wget https://s3.amazonaws.com/amazoncloudwatch-agent/amazon_linux/amd64/latest/amazon-cloudwatch-agent.rpm sudo rpm -U ./amazon-cloudwatch-agent.rpm # Configure agent sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-config-wizard

CloudWatch Alarms:

# Create high CPU alarm aws cloudwatch put-metric-alarm \ --alarm-name "HighCPU-i-12345" \ --alarm-description "Alarm when CPU exceeds 80%" \ --metric-name CPUUtilization \ --namespace AWS/EC2 \ --statistic Average \ --period 300 \ --threshold 80 \ --comparison-operator GreaterThanThreshold \ --evaluation-periods 2

Status Checks:

System Status Check

AWS infrastructure

Network, power, hardware

Action: Stop/start instance

Instance Status Check

Guest OS and software

Network config, memory

Action: Reboot instance

EC2 Instance Recovery:

# Create recovery alarm aws cloudwatch put-metric-alarm \ --alarm-name "AutoRecover-i-12345" \ --metric-name StatusCheckFailed_System \ --namespace AWS/EC2 \ --statistic Maximum \ --period 60 \ --threshold 1 \ --comparison-operator GreaterThanThreshold \ --evaluation-periods 2 \ --alarm-actions "arn:aws:automate:region:ec2:recover"

Performance Insights:

  • Monitor CPU credits for T-series instances
  • Check network performance metrics
  • Track EBS volume performance
  • Monitor memory and swap usage
  • Set up log aggregation with CloudWatch Logs
Use CloudWatch Dashboards to visualize all your metrics in one place!
